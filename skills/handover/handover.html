The idea in one line

Add only the missing half — a curated-prompt generator — and reuse everything CCB already gives you (/continue to ingest, autonew to launch, the pane registry to drive). No tmux, no Zellij, no new runtime deps.

Key decisions

  • Reuse /continue for ingestion — zero new code; write the curated doc as the newest .ccb/history file.
  • Never use ctx-transfer to send — it's a raw transcript dump, the exact thing to avoid.
  • Mode-aware, never breaks — auto-chain only fires with a live claude pane; handover_send.py catches all exceptions → manual fallback.
  • Atomic write before any reset — a dropped reset never loses the handover.
  • Personal infra — hardcoded CCB_LIB + py -3; not publishable as-is.

The finding that shaped mode-awareness

project_id: bfae8a9d…54e9c
claude record: NONE   |   pane_id: None   |   providers: []

The working session ran direct, not in a CCB pane — a skill can't /new its own REPL without a registered pane. So direct mode is primary; the paned auto-chain is a bonus.

Caveats & follow-ups

  • Pilot the paned auto-chain under a real CCB pane; bump time.sleep(1.5) if the 2nd command drops.
  • Paned mode resets the current pane (intended handoff semantics).
  • Windows/CCB coupling hardcoded — parameterise before any publish.
  • .ccb/ must be gitignored wherever /handover//continue run.
  • Future: true cross-platform launch (wt new-tab claude -p … / claude -p) à la pchalasani.
