/handover — secure continuation brief

Hand the current session off safely: compose a curated continuation brief, save it atomically, then use the explicit /clear + /continue boundary.

Built 2026-06-04 Hardened 2026-06-27 Data-only No process launch Cross-platform

The idea in one line

Compose a curated, redacted brief, save it before resetting, then let the user cross the session boundary explicitly. The skill never creates editor tasks or launches a process.

Flow

1Compose a curated handover brief (Goal · Status · Decisions+why · Failed-approaches · Files+lines · Next step · Don't-do · Suggested skills). The only "thinking" step.
▼
2Atomic-write → ./.ccb/history/<utc>-handover.md  (newest-wins)
▼
3Report the explicit boundary → /clear, then /continue
▼
Fresh session attaches the newest brief and carries on from its Next step.

Key decisions

  • Reuse /continue — one existing ingest path is easier to audit than a second launcher protocol.
  • Never use ctx-transfer — it's a raw transcript dump, the exact thing to avoid.
  • Keep the boundary explicit — starting the fresh session remains a visible user action.
  • Atomic write before reset — an interrupted reset still leaves a recoverable brief.
  • Remain data-only — no editor configuration, process launch, shell parsing, or platform-specific fallback.

The findings that shaped it

Automatic continuation requires an external execution mechanism. Editor startup tasks and process launchers create security and portability costs that are unnecessary for a documentation workflow, so the public skill deliberately stops at the saved brief.

Caveats & follow-ups

  • Explicit continuation. The user runs /clear and /continue; the skill intentionally does not automate this boundary.
  • .ccb/ must be gitignored wherever /handover//continue run — added to this repo's .gitignore.
  • Redaction remains mandatory. Briefs must not contain credentials, personal data, or raw conversation logs.
